TeamNest was built on a Windows 11 laptop using VS Code, Git, Docker Desktop, and Python 3.12 / Node tooling, with Postman and FastAPI's auto-published `/docs` for API exploration. The frontend runs on Next.js (App Router) with React 19, TypeScript, Tailwind CSS, shadcn/ui on Radix primitives, Ant Design and NextAuth.js. The backend runs on FastAPI + Uvicorn with Pydantic schemas, persists to PostgreSQL 16 through SQLAlchemy with Alembic-versioned migrations, and serves real-time channels, DMs, group chats, presence and notifications over FastAPI WebSockets. The AI assistant is a RAG pipeline orchestrated with LlamaIndex, embedding into Pinecone under per-team `team-{team_id}` namespaces and generating answers through Groq's low-latency LLM inference. Media is offloaded to Cloudinary's CDN, transactional email goes through a managed provider, and Pro-plan subscriptions go through Stripe Checkout with signed webhooks. The whole stack is containerised with Docker and brought up locally with one `docker compose up --build`; production runs on Render (backend + managed PostgreSQL) and Vercel (frontend, zero-config).

The product was delivered across five sprints. The first established the authentication core — registration, email verification, login with rotating refresh tokens, logout, password reset, profile editing, presence and theming — and the JWT + bcrypt + refresh-token security model that every later sprint reused. The second made the product real-time, standing up the WebSocket manager and Cloudinary attachment pipeline, and shipping channel messaging with edits, threads, pins, search, file sharing, `@mentions` and the unified real-time notification feed. The third pulled the multi-tenant social and work layers together: organizations with invitations and a member directory, the team substructure with a flexible role/permission model, the task lifecycle from creation through subtasks to approval, plus DMs, group chats and the friends/block social graph. The fourth added cross-cutting platform capabilities — Stripe Pro subscriptions and cancellation with signed-webhook plan transitions, org-wide global search backed by the same Pinecone index, and a sensitive-action audit log with a reversible-undo flow. The fifth closed the build with the RAG AI assistant grounded in org context, uploaded documents and inline PDFs, plus the final polish needed to declare the product feature-complete.

Each sprint produced its own C4 component diagram, class diagram and key sequence diagrams alongside the UI screens, and each closed with a short review and retrospective. Recurring themes across the retrospectives were that infrastructure built early (WebSockets, Cloudinary, the role/permission model, the vector index) compounded across later sprints, and that the parts that consistently underran were the ones touching external systems — Stripe webhooks, Postgres full-text tuning and notification fan-out — which would benefit from earlier integration with CI in a follow-up iteration.
